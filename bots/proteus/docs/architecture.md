# Proteus 架构设计

> 目标：以原创代码接近并最终超越 `kc.mega.BeepBoop`。方法论的参照是
> `docs/beepboop-analysis.md`——吸收其公开发表的思想（路径冲浪、可学习嵌入的
> KNN、门控危险集成、不确定性感知决策、正确的 bullet shadow），但不复制其代码
> 与类结构。本文是 Proteus 的目标架构与演进路线的设计依据；里程碑拆分见
> `docs/roadmap.md`。

## 1. 设计原则

1. **每 tick 一条流水线**：感知 → 状态 → 决策 → 执行，单向数据流，按数据依赖
   排序，无跨层回调。
2. **不确定性感知**：所有命中率、分类判断都带置信区间，策略开关统一走「区间
   相交即可能成立」的接口（`HitRate.overlaps`），而不是比较点估计。
3. **学习优于手调**：可调结构（KNN 嵌入、集成权重、门控阈值）尽量离线学习后
   回读为常量；在线适应用滚动统计完成。
4. **模拟保真**：自身、敌人、波的前推全部按引擎物理精确模拟；移动侧把模拟出的
   未来状态发布给瞄准侧，保证「子弹出膛点」与「瞄准假定的出膛点」一致。
5. **得分是目标函数**：子弹威力、护盾、残局收割都以期望得分（而非伤害）为优化
   目标。

包根 `zen.proteus`。可选择的机器人外壳是 `zen.Proteus`（一行具体类），实现是
`zen.proteus.Proteus`（抽象类）加各子系统——与其他模块的「shell + impl」惯例
一致。

## 3. 每 tick 流水线

事件回调（`onScannedRobot` 等）只暂存事件；主循环 `tick()` 统一消费，顺序按
数据依赖排列：

1. **状态**：`GameState.onStatus` 快照自身；若本 tick 有扫描则 `onScan` 重建敌
   人状态并做开火检测（能量差落在 [0.0999, 3.0001] 且上 tick 枪热 ≈ 0，补偿
   我方子弹造成的伤害）。
2. **雷达**：有扫描走 infinity lock（过冲 π/32）；无扫描且丢锁超时则沿最后
   锁定方向全场重搜。
3. **策略**（M7 起）：只更新分类标志位，不写指令。
4. **模式分支**（M7 起）：护盾模式时 `Shielder` 接管车身、炮管与开火；否则
   依次 `Mover.move`（车身通道）→ `Aimer.aim`（炮管与开火通道）。移动先于瞄
   准，是为了让枪用上移动模拟发布的未来状态（M3 起）。
5. **提交**：`Controls.apply()` 按固定顺序发出本帧全部 `set*` 指令并返回
   `Bullet`（供波与阴影记账），主循环 `execute()` 提交给引擎。

## 4. 指令通道纪律

Robocode 的 `set*` 是「设置后持续生效」语义，且同 tick 多次赋值是覆盖语义。
Proteus 用 `Controls` 把纪律变成结构：

- 每个子系统只写自己的通道：`Radar` → `radarTurnRadians`，`Mover` →
  `bodyTurnRadians` / `ahead` / `maxVelocity`，`Aimer` → `gunTurnRadians` /
  `firePower`；护盾模式时这些通道由 `Shielder` 独占。
- 通道为 null 表示本 tick 不触碰，上一条指令继续生效（雷达持续扫描依赖这一
  点）。
- `apply()` 是唯一调用 `set*` 的地方，顺序固定；主循环随后 `execute()`。

这样「模拟的控制」与「执行的控制」天然同源：移动模拟器预测的就是这帧指令的
结果。

## 5. 状态层

`GameState` 是双方状态的唯一来源，子系统不读机器人 API：

- 敌人位置由扫描重建（我方位置按 bearing/distance 投影），敌人枪热按冷却速率
  估计、开火检测命中时重置。
- 开火检测的误差来源（撞墙伤害、我方命中造成的能量扰动）逐项补偿；我方命中
  由 `noteOurBulletHit` 挂账，下一次扫描时冲销。
- 历史用定长环形缓冲保存（约 100 tick），供特征计算（近 10/20 tick 位移、变
  速计时器等）。
- M3 起增加**未来状态对账**：移动侧发布预测状态，下一 tick 用真实值比对，偏
  差超阈值即作废重建，预测不带病使用。

## 6. 子系统演进要点

### 6.1 移动：从轨道到路径冲浪（M2–M3）

- M2 先建波体系：敌方开火建波，精确交集求「本 tick 会被击中的 GF 区间」，访
  问统计做经验危险，真冲浪（前进/后退/停车三选一）。
- M3 实现了路径冲浪（方向序列上的 best-first 搜索、计划复用、节点预算）与
  bullet shadow（含跨 tick 修正）。**实测负结果**：路径搜索在粗糙经验危险下
  输给了三选一冲浪——它把噪声当信号，专挑模型里的窄缝穿，还制造低速窗口。
  结论：搜索机械保留（`PathSurfer`，行为级单测），默认引擎用三选一，M5 危
  险集成落地后 A/B 重评。阴影则确认有效并常驻（`BulletShadows`）。
- 与 BeepBoop 的差异：搜索用节点预算代替固定深度，天然免疫 skipped turn；
  这一机制保留在 `PathSurfer` 中备用。

### 6.2 危险评估：门控 + 动态权重集成（M5）

- 模型分三类：模拟枪（HOT、线性、圆形、均值线性、当前 GF，各自在对应 GF 涂
  窄峰）、KNN 危险模型（通用 + 按强对手离线训练的专家）、flattener（把「自己
  去过的位置」记为危险）。
- 每个模型带激活区间：敌方命中率置信区间与之相交才激活（敌枪越准，激活的模
  型越高级）；预测准确度（真实弹着点处的危险值，滚动平均）动态调权。
- 组合管线：各模型 151 bin 归一化 → 加权求和 → 指数核平滑 → 叠 bullet
  shadow 折扣 → 再归一化。
- 另设**反模式匹配模拟器**：以敌为观察者重演我方运动符号串，推断 PM 枪的提
  前量角并涂危险。

### 6.3 瞄准：KNN 枪组（M4）

- 首次开火后每 tick 造虚拟波，训练数据密度远超实弹；`virtuality` 特征区分
  「带弹波 / 离带弹波多远」。
- 两把 KNN 枪（主枪 / 反冲浪枪）按**我方**命中率区间硬切换：打不中（区间与
  低命中率带相交）说明对手会躲，换反冲浪枪。两把枪始终全量训练，切换零成本。
- 反冲浪的关键是 `didHit` 标记：命中后对方会避开被打中的角度，查询时带
  `didHit = 0` 把「命中之后」的样本挤出近邻。
- 选点用扫描线算法求邻居 GF 区间的加权密度峰（等价盒核 KDE 的 argmax）。
- 瞄准用波以**预测的下一 tick 我方位置**为波源（子弹真正的出膛点），炮管摆
  角用同一预测状态。

### 6.4 KNN 核心（M4，自研）

- 自研 KD 树（曼哈顿距离）；数据量小时退化为线性扫描，保证正确性优先。
- 嵌入函数逐维变换，参数离线学习（M8）后回读为常量；邻居数随数据量自适应；
  softmax 按距离加权。
- 瞄准模型存「能命中的 GF 区间」，移动模型存标量 hitGF。

### 6.5 子弹威力与主动挡弹（M6）

- 威力选择对期望得分建模：伤害率、命中返还、开火耗能、胜率估计（正态近似二
  项），选期望得分最大者；涌现行为（落后搏命、残血恰好击杀、贴脸全开）应自
  动出现，不写手工规则。
- 主动 bullet shadowing：开火瞬间评估十余个候选角，评分 = 命中概率 ÷ 阴影
  后危险的某次幂；敌人越准、弹越重，阴影收益权重越高。
- bullet shadow 计算遵循已发表的「correct bullet shadowing」思想：除标准
  (t, t) 阴影外补两个半权重的跨 tick 阴影。

### 6.6 对手分类与护盾（M7）

- 分类器全部在线、不确定性感知，不需要知道对手名字：ram（靠近速度 EMA）、
  镜像（与中心对称点距离 EMA）、HOT（默认假设，被非正前方弹命中后撤销）、
  冲浪者 vs 普通（我方命中率区间）。
- 子弹护盾对瞄准可预测的对手整体接管移动与瞄准：多个子弹预测器在线竞争识
  别敌人瞄准法，对「wiggle × 威力」组合精确计算拦截点。护盾名单（若做）来自
  离线评估，运行时满足退出条件（长期无弹、护盾失败）即转正常模式。

## 7. 与 BeepBoop 的关系

采用的**思想**（均为作者公开发表）：路径空间的 best-first 搜索、预计算路径
库、门控 + 动态权重的危险集成、命中率区间的策略切换、虚拟波训练、didHit 反
冲浪、主动 bullet shadowing、期望得分威力选择、离线学习嵌入参数。

我们的**不同**：

1. 指令帧结构（`Controls`）把单写者纪律显式化，子系统不可越权。
2. 路径搜索用迭代加深 + tick 预算，代替固定深度，天然免疫 skipped turn。
3. KD 树、嵌入、物理模拟全部自研实现，接口按 Kotlin 不可变数据类型设计。
4. 命中率区间用 Wilson score（Wald 在 p=0/1 退化，会假自信）。
5. 基准与诊断是一等公民：每个里程碑有量化验收（见 `docs/roadmap.md`），回
   合诊断与 A/B 覆盖从早期就内建。

## 8. 验证

- 单元测试：核心数学与状态逻辑（`core`、`state`）用 kotlin.test，不依赖引擎。
- 战斗验证：每个里程碑按 `docs/roadmap.md` 指定的 catalog 跑 `just duel`，记
  录 APS / PWIN / survival。
- 终极目标对照组：`top` catalog（含 BeepBoop 本体）。
